# eda009: 辞書入力付加の実現可能性調査

## 目的
BabyFST(Sahala 2020)の知見「80%の語が多義」を受けて、eBL_Dictionary / OA_Lexiconでの辞書引き結果を入力に付加する「前処理としての辞書活用」の実現可能性を調査。exp007で後処理としてのOA_Lexicon活用は中立だったため、入力側での活用を検討。

## 分析概要
- OA_Lexicon (39,332エントリ) とeBL_Dictionary (19,215エントリ) の構造・カバレッジを分析
- train翻字の全トークン (85,257個) に対する辞書マッチ率を計測
- 入力拡張フォーマット `token[lexeme]` のシミュレーションを実施
- 多義性・未マッチパターン・入力長増加の影響を評価

## 発見事項

### 1. OA_Lexiconのカバレッジは非常に高い

| 指標 | 値 |
|------|-----|
| ユニークトークンマッチ率 | **70.2%** (8,097/11,526) |
| 出現頻度ベースマッチ率 | **86.3%** (73,583/85,257) |
| マッチ内訳 - word | 6,203トークン |
| マッチ内訳 - PN（人名） | 2,182トークン |
| マッチ内訳 - GN（地名） | 107トークン |

**→ 翻字トークンの86%にlemma/タイプ情報を付加可能。**

### 2. eBL_Dictionaryのマッチ率は0%

- eBL_Dictionaryのword列は `lemma形式`（例: "abāku I"）で格納されており、翻字形式（例: "a-ba-ak"）との直接マッチができない
- OA_Lexiconが`form`列（翻字形式）→`lexeme`列（辞書見出し）の対応を持つのに対し、eBLはlemmaのみ
- **→ eBLは直接使えないが、OA_Lexicon経由でeBLの定義を引ける（lexeme→eBLリンク列あり）**

### 3. 多義性は限定的（BabyFSTの80%とは大きく異なる）

| 指標 | 値 |
|------|-----|
| 多義的トークン | 352/8,097 (**4.3%**) |
| 2 lexemes | 320トークン |
| 3+ lexemes | 32トークン |

**高頻度の多義的トークン:**
- `a-na` (freq=3,922): anna(word) / ana(word) / Anna(PN)
- `ma-na` (freq=2,452): manā'u(word) / Mana(PN)
- `i-na` (freq=1,265): Innā(PN) / ina(word)
- `igi` (freq=1,127): maḫar(word) / šēbu(word)

→ 多義性は主に「一般語 vs 固有名詞」の区別に集中。BabyFSTの80%多義との差は、OA_Lexiconが既にOld Assyrian方言に特化しており、活用形→lemmaの対応が確定的に登録されているため。

### 4. 入力拡張シミュレーション

**フォーマット例:**
```
Original:  KIŠIB ma-nu-ba-lúm-a-šur DUMU ṣí-lá-{d}IM
Augmented: KIŠIB[kunukku] ma-nu-ba-lúm-a-šur[PN] DUMU[mer'u] ṣí-lá-{d}IM
```

**全trainでの拡張統計:**
| 指標 | 値 |
|------|-----|
| OA_Lexiconマッチ | 83.9% |
| 未マッチ | 16.1% |
| **入力長増加率** | **平均1.67倍** |
| 512バイトtruncation（拡張前） | 40.4% |
| 512バイトtruncation（拡張後） | **65.5%** |

### 5. 最大の課題: 入力長増加によるtruncation悪化

- 辞書情報付加で入力が**平均1.67倍**に膨張
- 512バイト上限では truncation率が40.4%→65.5%に悪化
- **ByT5のバイトレベルでは辞書付加のコストが高すぎる**
- max_length=1024以上にするか、辞書情報のコンパクトな表現が必要

### 6. 未マッチトークンの内訳

高頻度未マッチの大部分は**数値**:
- `1` (1,039回), `2` (679回), `5` (404回), `10` (369回), `0.5` (285回) 等
- `a-limki` (365回): 地名の上付き表記
- `šu-den.líl` (57回): 神名含み固有名詞のdiacritic表記差

→ 数値は辞書引き不要（規則的に翻訳可能）。実質的な未マッチは少ない。

### 7. Testデータでのマッチ率

| 指標 | 値 |
|------|-----|
| OA_Lexiconマッチ（ユニーク） | 49.2% (29/59) |
| マッチ（頻度ベース） | 57.6% (49/85) |

※testデータは4件のダミーのため参考値。本番テスト（~4,000文）では異なる可能性。

## 実験への示唆

### 実行可能なアプローチ

1. **コンパクトな辞書タグ付加**
   - 全lexemeではなく、**PNタグのみ付加**: `token[PN]` (入力増加を最小化)
   - 人名は翻訳で最も誤りやすい（Gutherz 2023）ため、PN/GN区別だけでも価値あり
   - 入力増加: PN/GN該当トークンのみなので+10-15%程度に抑えられる

2. **ロゴグラム→lemma変換の前処理**
   - `KIŠIB`→`kunukku`, `DUMU`→`mer'u`, `KÙ.BABBAR`→`kaspu` 等
   - ロゴグラム（大文字表記）は意味が確定的で多義性なし
   - 入力長を増やさずに情報量を増やせる

3. **辞書情報を別チャネルで入力**（高難易度）
   - 翻字テキストとは別に、辞書引き結果を第2入力として与える
   - cross-attentionで辞書情報を参照するアーキテクチャ変更が必要

### 推奨しないアプローチ

- **全トークンにfull lexeme付加**: 入力1.67倍でtruncation悪化が深刻
- **eBL_Dictionary直接使用**: 翻字形式とのマッチ不可能（OA_Lexicon経由が必要）

### 実装の優先順位

1. PN/GNタグ付加（低コスト、高効果見込み）→ **まず実験すべき**
2. ロゴグラム→lemma変換（入力長不変、情報量向上）
3. 多義的トークンの曖昧性解消（a-na, ma-na等のPN/word区別）
