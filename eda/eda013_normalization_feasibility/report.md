# EDA013: 入力翻字の正規化対象の実態調査

## 分析の目的
LB 35.5ノートブック(ngyzly)が実装する入力正規化が、我々のデータにどれだけ当てはまるかを定量的に確認する。

## 分析概要
train(1,561件) + test(4件) + additional(1,165件) = 2,730件の翻字データに対して、正規化対象パターンの出現頻度を調査した。

## 発見事項

### 1. ASCII→Unicode変換: 対象なし
- `sz→š`, `s,→ṣ`, `t,→ṭ`, `vowel+2/3` いずれもゼロ
- **データは既にUnicode化済み。この正規化は不要。**

### 2. ギャップマーカー: 1,118/2,730 docs (41.0%)
- ほぼ全て `<gap>` 形式で統一済み（4,827回出現）
- testに1件だけ `…`（三点リーダー）が存在
- **既に統一済みのため、`…`→`<gap>` の1件対応程度。効果は極めて限定的。**

### 3. 小数/分数: 816/2,730 docs (29.9%)
- 分数の内訳: 1/3=833回, 2/3=552回, 5/6=232回, 1/6=70回
- train: 404件(25.9%), additional: 412件(35.4%)
- **ByT5はバイトレベルなので、`0.33333`→`⅓` の変換はトークン数削減になる**
- ただしtest=0件なので、推論時には影響しない。学習時の効率改善のみ。

### 4. 限定詞: 3 docs のみ
- 大文字(ḪI): train 1件, 小文字(ki): test 2件
- **ほぼゼロ。この正規化は不要。**

### 5. 特殊文字: 2,520/2,730 docs (92.3%)
- `ḫ/Ḫ`: train 1,313件(4,935回), additional 1,075件(4,977回) — **非常に多い**
- 下付き数字 `₀-₉`: train 1,173件(3,668回) — **多い**
- `ʾ`(アレフ): 1件, `ₓ`: 4件 — ごく少数

**重要な考察**: `ḫ→h` と `₀→0` は見た目の変更は小さいが、ByT5にとってはバイト列が変わる。
- `ḫ`(U+1E2B)はUTF-8で3バイト → `h`(U+0068)は1バイト
- `₀`(U+2080)はUTF-8で3バイト → `0`(U+0030)は1バイト
- **これにより入力トークン長が大幅に短縮される可能性がある（ByT5のボトルネック緩和）**

### 6. トークン正規化のインパクト
- ユニークトークン: 18,510 → 18,228 (282削減, 1.5%)
- マージされるグループ: わずか7組（下付き数字と通常数字の混在）
- **語彙レベルでの統合効果は小さい**

### 7. train vs test の分布差
- testに `…` が1件あり、trainには存在しない（test固有パターン）
- ただしtest=4件と極小なので、これだけでは判断できない

## 結論: 正規化の優先度

| 正規化 | 対象件数 | 効果見込み | 推奨 |
|--------|---------|-----------|------|
| ASCII→Unicode | 0件 | なし | 不要 |
| ギャップ統一 | ほぼ統一済み | ほぼなし | 不要 |
| 限定詞正規化 | 3件 | なし | 不要 |
| 分数正規化 (0.33→⅓) | 816件(30%) | 中（学習時のトークン削減） | 検討 |
| `ḫ→h`, `Ḫ→H` | 2,388件(87%) | **大（バイト長削減）** | **要検討** |
| 下付き数字→ASCII | 1,173件(43%) | **大（バイト長削減）** | **要検討** |

### 注意点
- `ḫ→h`は**情報の損失**を伴う。ḫ(voiceless pharyngeal fricative)とh(glottal fricative)は異なる音素
- 下付き数字もアッカド語学では意味的に区別される（ホモフォンの識別番号）
- LB35.5ノートブックのモデルは「潰した」状態で学習済みなので正規化が効くが、我々のデータは元々整然としているため、**潰すことで失われる情報のデメリットの方が大きい可能性がある**
